## 1. Necessity and Purpose of the Assessment

### 1.1 Why this DPIA is Required

The Quality Governance Platform processes personal data of employees, contractors, and third parties
in the context of health and safety incident reporting, audit management, corrective action tracking,
risk management, and regulatory compliance. Processing includes:

- Special category data: health and injury information within incident reports
- Processing on behalf of multiple organisations (multi-tenant SaaS)
- Automated processing for compliance scoring and risk trend analysis
- Cross-border data processing considerations (Azure cloud infrastructure in UK South / West Europe regions)

Under Article 35(1) UK GDPR, processing likely to result in a high risk to data subjects requires a DPIA.
The combination of special category health data, large-scale multi-tenant processing, and automated
risk scoring meets the Article 35(3) threshold.

### 1.2 Scope

| Dimension | Scope |

## 3. Necessity and Proportionality Test

### 3.1 Necessity

Each data category processed is necessary for the stated purpose:

| Data Category | Necessity Justification | Could Less Data Achieve the Same? |
|---------------|------------------------|-----------------------------------|
| Injury description | Legally required for RIDDOR reporting; risk assessment | No — regulators require description |
| Reporter name + contact | Required for regulatory follow-up; witness identification | Partial pseudonymisation applied for read-only views |
| Assignee email on CAPA | Required to route notifications; track accountability | No — accountability is core to CAPA purpose |
| Vehicle registration | Required for RTA legal obligations | No — insurer and police require reg. |
| Audit trail (user ID) | Required for security and accountability | User IDs not names — already minimised |

### 3.2 Proportionality

- **Retention periods** are capped: operational logs 90 days; incident data 7 years (aligned with UK H&S legal minimums). See `docs/compliance/retention-automation-evidence.md`.
- **Access controls**: tenant-isolated ABAC model; users only access their own tenant's data.
- **Data minimisation**: forms only collect fields required by the relevant workflow. Field-level config is managed via `AdminConfig`.
- **Pseudonymisation**: audit trail entries reference `user_id` not names; cross-reference to profile only with explicit authorisation.

## 4. Risk Assessment

### 4.1 Risk Matrix

| Risk ID | Risk Description | Likelihood (1-5) | Impact (1-5) | Inherent Score | Mitigation | Residual Score |
|---------|-----------------|-----------------|--------------|----------------|------------|----------------|
| R01 | Unauthorised access to personal data (external breach) | 2 | 5 | 10 | Azure WAF, TLS 1.2+, JWT auth, ABAC, rate limiting, Trivy/DAST CI gates | 4 |
| R02 | Insider access to cross-tenant PII | 2 | 5 | 10 | Tenant isolation middleware, ABAC, audit trail, row-level filtering | 4 |
| R03 | Data retention beyond permitted period | 2 | 3 | 6 | Automated retention enforcement (`retention-automation-evidence.md`), quarterly audit | 2 |
| R04 | Data subject rights request not fulfilled | 2 | 3 | 6 | DSR workflow in admin panel; 30-day SLA tracked in CAPA | 2 |
| R05 | Breach notification delay > 72 hours | 2 | 4 | 8 | Incident response runbook (`INCIDENT_TEMPLATE.md`), Azure Monitor alerts | 3 |
| R06 | Third-party processor non-compliance | 2 | 4 | 8 | Azure DPA in place; no other sub-processors outside Microsoft | 3 |
| R07 | Accidental exposure via API (misconfigured endpoint) | 2 | 4 | 8 | OpenAPI contract tests, DAST advisory scan, Schemathesis property tests (CI) | 3 |
| R08 | AI/GenSpark processing of PII without consent | 1 | 5 | 5 | GenSpark API called with anonymised finding IDs only; no PII in prompt | 1 |

### 4.2 Overall Risk Conclusion

Residual risk scores are all ≤ 4 (Low–Medium). The processing can proceed. No residual High risks remain
after mitigation. Continued monitoring via Azure Monitor alerts and quarterly DPIA review is required.

## 5. Measures to Address Risks

| Measure | Implementation | Evidence |
|---------|---------------|----------|
| Encryption at rest | Azure PostgreSQL transparent data encryption; Blob Storage CMK | Azure portal encryption settings |
| Encryption in transit | TLS 1.2+ enforced; HSTS header max-age=31536000 | `src/main.py` SecurityHeadersMiddleware |
| Access control | Azure AD + JWT; ABAC role model; tenant isolation | `src/api/dependencies/tenant.py` |
| Audit logging | All write operations logged with user_id, timestamp, action | `src/domain/services/audit_log_service.py` |
| Breach detection | Azure Monitor alerts; 5xx rate alerting | `.github/workflows/deploy-production.yml` monitoring step |
| Data minimisation | Form field config controls what is collected per workflow | `src/api/routes/form_config.py` |
| Retention automation | Automated deletion after retention period | `docs/compliance/retention-automation-evidence.md` |
| DAST security scanning | Advisory OWASP ZAP scan on every PR | `.github/workflows/ci.yml` (advisory) |
| PII inventory | Automated scan of models on every PR touching models | `scripts/governance/audit_pii_fields.py` |
| Schemathesis API testing | Property-based API testing to catch data leakage | `tests/integration/test_schemathesis_api.py` |

## 8. Decision and Outcome

**Decision**: Processing may proceed.

**Rationale**:
- All processing has an identified lawful basis under UK GDPR Article 6 (and Article 9 where applicable).
- Processing is necessary for the stated purposes and proportionate.
- Residual risks have been reduced to Low–Medium through the identified technical and organisational measures.
- No high-residual risks remain.
- ICO prior consultation is not required.

**Conditions**:
1. Quarterly review of this DPIA, triggered by any material change to the processing.
2. PII inventory (`scripts/governance/audit_pii_fields.py`) must run on every PR touching models.
3. Retention automation evidence must be updated annually.
4. Breach response runbook must be tested annually (tabletop drill).

# Full-System Forensic Findings Register

## Scope
This register captures a code-and-workflow forensic review of the platform across:

- backend API, domain services, models, tasks, and middleware
- frontend routing, auth/session handling, page-level integrations, and API clients
- CI/CD, deployment verification, observability, and production-safety controls

The review is evidence-led and based on repository state. It does not claim live Azure portal state unless that state is encoded in repo workflows or directly queried during the review.

## Integration Map
The dominant architecture pattern is:

`frontend page -> frontend api/client.ts group -> FastAPI route -> domain/service layer -> SQLAlchemy model/storage/task -> deploy/runtime guard`

The main forensic issue pattern is not absence of modules. It is partial integration:

- routes, models, or pages exist
- but persistence, auth contract, route mounting, storage wiring, or tests are incomplete
- and some frontends compensate with local mock/default transforms

## Risk Register

| ID | Severity | Risk | Evidence | Why it matters |
|---|---|---|---|---|
| `RISK-001` | `P2` | AI/document configuration drift | `src/domain/services/document_ai_service.py`, `src/core/config.py`, `requirements.txt` | The document AI service is written around Anthropic, the typed settings file does not expose a first-class Anthropic setting, and backend dependencies include `google-generativeai`. This suggests configuration and provider drift that can complicate deployment and support. |
| `RISK-002` | `P2` | Auth duplication across admin and portal flows | `src/api/routes/auth.py`, `frontend/src/utils/auth.ts`, `frontend/src/contexts/PortalAuthContext.tsx`, `frontend/src/api/client.ts` | Admin and portal auth are centralized better than before, but they still use different storage models and restore logic. The portal context explicitly documents half-authenticated states when platform tokens are missing. |
| `RISK-003` | `P2` | Release controls are strong but operationally brittle | `.github/workflows/ci.yml`, `.github/workflows/deploy-staging.yml`, `.github/workflows/deploy-production.yml`, `docs/evidence/release_signoff.json` | The release system is mature, but the platform depends on many gates, artifacts, and Azure-side moving parts. Any module drift that escapes tests can become expensive at deploy time because verification is strict and late. |
| `RISK-004` | `P3` | Frontend environment mismatch is detected but non-blocking | `frontend/src/config/apiBase.ts` | The frontend detects API/frontend environment mismatches, but it returns `null` on validation failures and does not hard-block. This is safe for uptime but weak for strict environment integrity. |
| `RISK-005` | `P1` | UAT override authorization trusts a caller-controlled user header | `src/core/uat_safety.py`, `src/core/middleware.py` | The production UAT write override can derive the authorizing user from `X-User-ID` when `request.state.user_id` is absent, but the shared request-state middleware only sets `request_id`. That creates a spoofable authorization path for a sensitive production control. |
| `RISK-006` | `P1` | Document library tenant isolation is not enforced end-to-end | `src/api/routes/documents.py`, `src/domain/models/document.py` | The document model has `tenant_id`, but upload does not set it and list/get queries do not scope by tenant. This creates a cross-tenant document exposure risk once users can enumerate or guess document IDs. |
| `RISK-007` | `P2` | RBAC permission checks rely on substring search over serialized JSON | `src/domain/models/user.py` | `User.has_permission()` tests `permission in role.permissions` where permissions are stored as a text JSON blob. That can produce false-positive matches and weakens confidence in fine-grained authorization decisions. |
| `RISK-008` | `P2` | Telemetry ingestion is unauthenticated and accepts arbitrary payload logging | `src/api/routes/telemetry.py`, `src/core/uat_safety.py` | Telemetry POST endpoints are public, web-vitals accepts an untyped dict payload, and the handler logs the payload while always returning success. This expands log-noise, PII, and abuse risk on a path explicitly exempted from UAT write blocking. |
| `RISK-009` | `P2` | Employee portal submissions are pinned to fallback tenant `1` | `src/api/routes/employee_portal.py`, `src/core/config.py` | Portal-created incidents, complaints, and RTAs all use `DEFAULT_PORTAL_TENANT_ID = getattr(settings, "default_tenant_id", 1)`, but the typed settings model does not define `default_tenant_id`. That means multi-tenant portal traffic can silently collapse into tenant `1`. |
| `RISK-010` | `P2` | Portal and frontend API base handling is inconsistent | `frontend/src/pages/PortalTrack.tsx`, `frontend/src/config/apiBase.ts`, `frontend/src/api/client.ts` | Portal tracking mixes relative `/api/...` fetches with `API_BASE_URL` fetches, while the shared Axios client force-rewrites non-HTTPS base URLs. This creates environment-dependent failures, especially across local/dev and reverse-proxy assumptions. |
| `RISK-011` | `P2` | Portal can present a signed-in state without a working platform token | `frontend/src/contexts/PortalAuthContext.tsx` | The portal restores `user` from local storage even when no `platform_access_token` exists, and demo login creates a user session without one. That produces half-authenticated UX where report submission may appear available but token-backed features cannot work reliably. |
| `RISK-012` | `P2` | Offline sync can silently discard unsent changes | `frontend/src/lib/syncService.ts` | Failed offline records are retried a fixed number of times, then deleted without user-visible escalation, while periodic sync suppresses errors. That creates a latent data-loss risk for queued writes. |
| `RISK-013` | `P3` | Test auth fixtures can mask authentication failures | `tests/conftest.py` | Test token fixtures swallow exceptions and return empty auth headers, so parts of the suite can execute unauthenticated without failing setup. This weakens the signal for auth regressions. |

## Recommended Serialized Fix Queue
This is not implementation approval. It is the recommended production-safe sequence implied by the forensic review.

1. `FIX-001` Stabilize stale auth/user-contract route families (`workflow`, `kri`, `policy_acknowledgment`, `rca_tools`) and verify admin operational surfaces work with the current dependency model.
2. `FIX-002` Replace mock/local-only compliance evidence flow with persisted API-backed evidence links and reporting.
3. `FIX-003` Mount and wire cross-standard mappings / IMS unification into the live application surface.
4. `FIX-004` Complete audit report/document ingestion with durable storage, extraction, and downstream action-mapping hooks.
5. `FIX-005` Reconcile Planet Mark and UVDB backend readiness with frontend behavior and restore meaningful integration coverage.
6. `FIX-006` Repair and validate the production UAT override control path.

## Conclusion
The platform is broad and ambitious, with substantial real implementation in routing, models, release controls, and test infrastructure. The principal forensic conclusion is that the system is not uniformly integrated end-to-end across all advertised governance functions. The highest-impact defects are contract drift and partially wired capability, not total absence of capability.
